1. Architecture globale

L'application repose sur une architecture trois-tiers classique, orchestrée localement via Docker Compose. Chaque couche est isolée dans son propre conteneur, communique via un réseau Docker interne, et est indépendamment scalable en production.

NAVIGATEUR WEB — Client React
React 18  ·  React Query  ·  Recharts  ·  Axios  ·  Zod
▼   HTTPS / REST   ▼
API GATEWAY — Nginx (reverse proxy)
Routage  ·  TLS termination  ·  Rate limiting  ·  CORS headers
▼   HTTP interne   ▼
BACKEND — FastAPI (Python 3.12)
Pydantic  ·  SQLAlchemy 2 async  ·  Alembic  ·  Celery Worker  ·  JWT RS256
▼ asyncpg / SQLAlchemy     ▼ Redis pub/sub (Celery)     ▼ HTTP (Frankfurter, SMTP)
PostgreSQL 16
Base de données principale
Redis 7
Cache applicatif + Broker Celery
Services externes
Frankfurter API (taux) · SMTP (e-mails)

2.3 Authentification (JWT RS256)

1. POST /api/v1/auth/login  { email, password }
       └─ FastAPI vérifie le hash bcrypt (cost 12) en base
       └─ Génère access_token  (JWT RS256, TTL 15 min)
       └─ Génère refresh_token (JWT RS256, TTL 7 jours)
       └─ refresh_token persisté en base (table refresh_tokens, hash SHA-256)

2. Stockage côté client
       access_token  → mémoire JavaScript (variable d'état, non persisté)
       refresh_token → cookie HttpOnly Secure SameSite=Strict

3. Requêtes authentifiées
       Authorization: Bearer <access_token>
       FastAPI middleware : décode JWT, injecte user_id dans request.state

4. Renouvellement
       À expiration de l'access_token : POST /api/v1/auth/refresh
       → nouveau access_token si refresh_token valide
       → 401 Unauthorized + redirect login si refresh_token expiré/révoqué

5. Déconnexion
       POST /api/v1/auth/logout
       → refresh_token marqué revoked_at en base
       → client efface l'access_token de la mémoire

4. Diagrammes d'états

4.1 Cycle de vie d'une dépense récurrente

La gestion des récurrences est pilotée par Celery beat (tâche périodique toutes les heures). La transition entre états est le seul automate métier non trivial de l'application.

⬜ INITIALE — Utilisateur crée une dépense avec récurrence
▼  → next_due_at calculée selon la fréquence choisie
✅ ACTIVE — next_due_at = date de prochaine occurrence
▼  Celery beat atteint next_due_at → génère une expense + recalcule next_due_at
▼ Utilisateur suspend manuellement
⏸ SUSPENDUE
↕ Réactivation possible
▼ ends_at atteinte
🏁 TERMINÉE
État final non réversible
▼ Suppression soft (is_active = false)
🗑 SUPPRIMÉE — état final
Règle métier : modification d'une série

Lors de la modification ou suppression d'une dépense récurrente, l'utilisateur choisit entre :
(a) modifier/supprimer uniquement l'occurrence sélectionnée — la dépense est détachée de la série (recurrence_id → NULL)
(b) modifier/supprimer toute la série — la recurrence est mise à jour / is_active = false

4.2 Cycle de vie d'une session utilisateur

ANONYME ──[POST /auth/login — credentials OK]──────► AUTHENTIFIÉ
                                                        (access_token RS256, TTL 15 min)
         └─[credentials KO, 5 tentatives]──────► BLOQUÉ (lockout Redis, TTL 15 min)

AUTHENTIFIÉ ──[access_token expiré]──► RAFRAÎCHISSEMENT
                  ├─[refresh_token valide]──► AUTHENTIFIÉ (nouveau access_token)
                  └─[refresh_token expiré / révoqué]──► ANONYME

AUTHENTIFIÉ ──[POST /auth/logout]──► ANONYME
                  └─ refresh_token.revoked_at = NOW() en base
                  └─ client efface l'access_token de la mémoire

4.3 Cycle de vie d'une alerte budget

SOUS LE SEUIL ──[dépense ajoutée, cumul ≥ 80% du budget]──► ALERTE 80%
                  └─ Celery task : send_budget_alert_email (seuil: "attention")

ALERTE 80% ──[dépense ajoutée, cumul ≥ 100%]──► DÉPASSEMENT
                  └─ Celery task : send_budget_alert_email (seuil: "dépassement")

DÉPASSEMENT ──[nouvelle période / budget réinitialisé]──► SOUS LE SEUIL
             └─[utilisateur augmente le budget]──────────► SOUS LE SEUIL

5.1 Décisions sécurité

DomaineDécision retenue
Mots de passeHachage bcrypt, cost factor 12. Jamais stocké en clair, jamais journalisé dans les logs.
Tokens JWTRS256 asymétrique. access_token en mémoire JS (non persisté), refresh_token en cookie HttpOnly Secure SameSite=Strict.
Isolation donnéesuser_id systématiquement injecté depuis le token JWT par le middleware FastAPI — jamais depuis le body de la requête.
Rate limitingNginx : 10 req/s par IP sur /api/v1/auth/*. 100 req/s sur les autres endpoints API.
Headers sécuritéHSTS, X-Frame-Options: DENY, CSP strict, X-Content-Type-Options: nosniff — configurés dans nginx.conf.
CORSOrigin whitelist stricte en production. Aucun wildcard (*). Préflight OPTIONS géré par Nginx.
Brute forceLockout 15 min après 5 échecs de login consécutifs. Compteur stocké dans Redis (TTL 15 min, clé par IP + email).

Points d'attention pour la production

1. Secrets : utiliser Docker secrets ou un vault (HashiCorp Vault) — pas de secrets en clair dans les variables d'environnement committées.

2. Backups PostgreSQL : pg_dump programmé via cron, chiffré (GPG) et stocké hors du serveur applicatif (S3 ou équivalent).

3. Monitoring : Prometheus + Grafana ou service managé. Alertes sur latence API > 500 ms et taux d'erreur 5xx > 1%.

4. Migrations Alembic : appliquées automatiquement au démarrage du conteneur backend via entrypoint.sh — avec vérification de santé de la base avant démarrage de l'API.
